# 窄屏顶部菜单换行设计

## 目标

修复选择“顶部菜单模式”后，在手机/平板窄屏中顶部菜单与账本、日期、退出控件争抢同一行空间，导致菜单项被挤压的问题。窄屏下菜单应独占头部第二行，位于顶部控制行和多页签之间；桌面端保持现有单行布局。

## 范围与约束

- 仅影响 `horizontal`（顶部菜单）布局模式。
- 使用项目现有的 `lg=1024px` 响应式边界：宽度小于 1024px 时启用第二行菜单。
- 垂直、垂直混合、顶部菜单混合模式不改变。
- 不改变菜单数据、路由跳转、账本选择、日期显示或退出登录逻辑。
- 保留工作区已有的 `.env.development` 修改，不将其纳入本次提交。

## 方案

### 结构

`GlobalHeader` 拆分为两个视觉区域：

1. 顶部控制行：保留 Logo、菜单/面包屑、账本选择、日期和退出按钮。桌面顶部菜单继续放在这一行。
2. 窄屏菜单行：仅当 `showHeaderMenu` 为真且窗口宽度小于 1024px 时显示，独占整行并渲染现有 `HeaderMenu`。

`HeaderMenu` 仍使用现有的 `n-menu`、`mode="horizontal"` 和 `min-w-0`，仅改变其父级位置和可用宽度。

### 布局高度

`AdminLayout` 使用固定的 `header-height` CSS 变量来定位头部、标签页和内容，因此仅移动 DOM 不足以完成修复。`BasicLayout` 根据同一窄屏条件计算头部高度：

- 顶部菜单模式且宽度小于 1024px：`theme.header.height * 2`
- 其他情况：`theme.header.height`

`GlobalHeader` 的两个行容器各自使用 `theme.header.height`，确保实际头部高度与 `AdminLayout` 的布局高度一致。主题头部高度变化时，两行会同步变化。

### 响应式更新

使用 `@vueuse/core` 的窗口宽度响应式值，使窗口旋转、开发者工具调整或桌面窗口缩放时，菜单位置和 `AdminLayout` 高度同步切换。窄屏判断只在 `horizontal` 模式下影响布局。

## 验证标准

- 在宽度小于 1024px 且选择顶部菜单模式时，第一行的账本、日期、退出控件保持可见，菜单完整出现在第二行。
- 第二行菜单与多页签之间没有重叠，页面内容从正确的头部高度开始。
- 在宽度大于等于 1024px 时，顶部菜单仍与现有桌面布局一致。
- 切换到垂直、垂直混合或顶部菜单混合模式时，头部高度和菜单行为保持原有逻辑。
- `typecheck` 和生产构建通过。

## 测试策略

为响应式判断和头部高度计算抽取一个纯函数，并先添加窄屏/桌面/非顶部模式的回归测试。模板结构通过类型检查和生产构建验证；最终使用浏览器或本地预览检查 655px 左右的窄屏布局，确认视觉位置符合需求截图。
